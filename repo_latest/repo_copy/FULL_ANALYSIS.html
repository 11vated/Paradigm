1. Architecture Overview

Paradigm is a two-tier generative platform built on the concept of seeds — genetic blueprints containing typed genes that can be bred, mutated, composed across domains, cryptographically signed, and grown into deterministic artifacts. The core innovation is the GSPL (Generative Seed Programming Language) and a physics-first approach to content generation that routes creative parameters through quantum field theory solvers before extracting 3D meshes, textures, and animations.

2. The Dual Backend Problem

Critical Architecture Issue

The project has two separate backends that overlap in functionality but diverge in capability. The TypeScript backend (Express, server.ts) has the QFT physics solvers and Gemini integration. The Python backend (FastAPI, paradigm/backend/) has the full GSPL parser, all 26 domain engines, the gene system, and MongoDB persistence. Neither is a superset of the other.

TypeScript Backend (Active)

  • Express server, 983 lines
  • 5 QFT solvers (EM, Dirac, QED, QCD, Gravity)
  • 4-stage pipeline (GSPL → QFT → Mesh → Texture)
  • Gemini AI integration (generate, mutate, embed)
  • ECDSA P-256 sovereignty
  • JSON flat-file persistence
  • Regex-based GSPL parser

Python Backend (Dormant)

  • FastAPI server, 618 lines + 6400 LOC modules
  • Full GSPL parser with AST + type checker (849 lines)
  • All 26 domain engines with real grow() methods
  • 17 gene types x 4 operators = 68 functions
  • 9 cross-domain functor bridges
  • MongoDB async persistence (Motor)
  • ECDSA P-256 sovereignty (ecdsa library)

The frontend currently connects to the TypeScript backend only. The Python backend represents a more complete implementation of the GSPL specification but lacks the QFT physics engine. These two backends need to be unified — the Python backend's language infrastructure with the TypeScript backend's physics engine.

3. TypeScript Core: QFT Physics Engine

The QFT module contains five genuine numerical solvers implementing real physics. These are not toy models — they use standard computational physics techniques found in academic codes.

Solver Method Lines Physics Depth
EM (Maxwell) FDTD with PML 246 Full 3D Maxwell equations, Yee leapfrog, Courant stability Real
Dirac Spinor evolution 254 4-component spinor, Pauli matrices, Coulomb potential with softening Real
QED Coupling Bidirectional ~100 Fermionic current J = psi-bar gamma psi drives EM, EM feeds back to Dirac Simplified
QCD (Lattice) Metropolis MC 244 SU(2) lattice gauge, Wilson action, plaquette observables, 4D periodic BC Rigorous

Key Finding

The QCD solver is the most sophisticated module in the entire codebase. It implements actual Monte Carlo lattice gauge theory with SU(2) matrix multiplication, staple calculation, and Metropolis accept/reject — the same algorithmic framework used in real lattice QCD research. The EM solver uses proper FDTD with Perfectly Matched Layer boundaries, which is standard in electromagnetic simulation.

GSPL Compiler (Domain → Physics Mapping)

The compiler maps creative domains to field types deterministically: character/organic → Dirac (fermionic), vfx/energy → QED (coupled), matter/prop → QCD (strong force), cosmos/environment → Gravity. Seed genes (core_power, stability, complexity) map to physics parameters (field amplitude, localization width, momentum). This mapping is the conceptual bridge between creative intent and physical simulation.

4. Python Core: 26 Domain Engines

Contrary to Initial Assessment

The project instructions estimated "likely 2-3 real engines, others are skeletons." After deep analysis: all 26 engines are implemented with real logic. Every grow() function extracts genes, applies mathematical transformations, and returns rich artifact descriptors with domain-specific render hints. No stubs, no TODOs, no placeholder returns.

The 26 engines span these domains, each with substantive grow() methods in engines.py (356 lines total):

Primary Engines (5)

character, sprite, music, visual2d, procedural — most detailed implementations with 30-40 lines each. Character engine computes body dimensions from strength/agility genes, derives HP/MP stats, extracts RGB from palette vectors.

Extended Engines (21)

fullgame, animation, geometry3d, narrative, ui, physics, audio, ecosystem, game, alife, shader, particle, typography, architecture, vehicle, furniture, fashion, robotics, circuit, food, choreography — each 10-20 lines with real gene extraction and domain-specific output.

Gene System (509 Lines, 68 Functions)

All 17 gene types implement all 4 operators (validate, mutate, crossover, distance). Notable implementations include quantum genes with amplitude normalization, graph genes with three mutation modes (weight, add edge, delete), temporal genes with ADSR envelope support, and sovereignty genes that are deliberately immutable (mutation and crossover are no-ops by design).

Composition Engine (277 Lines, 9 Functors)

Nine pre-registered functor bridges enable cross-domain transformation with semantic mappings. For example, character_to_music maps warrior archetype to minor mode, mage to dorian, rogue to blues; tempo derives from strength gene; timbre from agility. These are real creative transformations, not passthrough copies. BFS pathfinding enables multi-hop composition between any two domains.

5. GSPL Language & Parser

TypeScript Parser (gspl-parser.ts)

Regex-based, extracts seed blocks, parses key-value pairs with depth-aware nesting. Handles scalar/vector/categorical types. Functional but fragile. ~80% complete. Used by the active frontend.

Python Parser (gspl_parser.py, 849 lines)

Full recursive-descent parser with lexer, AST generation, and Hindley-Milner type system integration. Supports all 26 keywords, 17 gene types, seed operations (breed, mutate, compose, evolve, grow). Production-grade. Currently dormant.

Language Specification (from data/spec/04-gspl-language.md)

GSPL is formally specified with 26 keywords, ~30 operators, 6 literal types (int, float, string, bool, vector, nil), and a complete grammar supporting seed declarations, function definitions, type declarations, domain schemas, trait implementations, and GPU annotations. The spec defines 25+ AST node types and includes formal operational semantics for seed operations.

6. Sovereignty & Cryptography

Production-Grade Crypto

Both backends implement real ECDSA P-256 (secp256r1) signing. The TypeScript layer uses Node.js crypto with PEM-encoded keys. The Python layer uses the ecdsa library with JWK-formatted public keys. Seeds are content-addressed (SHA-256 of canonical JSON) before signing. Verification is stateless.

The sovereignty specification (data/spec/05-sovereignty.md, 241 lines) defines RFC 8785 JSON Canonicalization, SHA-256 hashing, and RFC 6979 deterministic ECDSA. Lineage signatures compose, meaning a child seed's signature chain includes its parents. This is the foundation for provenance tracking and eventual on-chain minting.

7. Intelligence Layer

The intelligence module (318 lines) integrates Google's Gemini API for three operations: embedding generation (text-embedding-004, 768 dimensions), seed generation from natural language prompts (gemini-2.5-flash), and AI-assisted mutation/breeding. All operations have deterministic fallbacks: pseudo-embeddings via SHA-based LCG, hash-based seed generation, and Gaussian/random mutation.

Dependency Risk

The platform currently relies on Gemini for its most user-facing features (prompt-to-seed, intelligent mutation). The fallback mechanisms work but produce semantically meaningless results. The vision document describes a native GSPL agent (fine-tuned Llama 3 70B with QLoRA + ChromaDB RAG) that would eliminate this dependency, but it has not been implemented.

8. Asset Pipeline & The Sphere Problem

Root Cause: The Sphere Problem

The mesh extractor claims to implement Marching Cubes but actually performs naive voxel cube generation. For each voxel above threshold, it generates 6 faces as axis-aligned boxes. There is no edge interpolation, no vertex interpolation between grid points, and no smooth isosurface extraction. If the QFT field is a Gaussian (spherically symmetric), the output is a cubic shell, not a smooth sphere. This is why everything looks the same — the extraction algorithm cannot distinguish between different field topologies at the visual level.

Stage Module Quality Issue
1. Compile gspl/compiler.ts Good Deterministic domain → QFT mapping. Clean.
2. Simulate qft/*.ts Good Real PDE solvers. Output is genuine 3D field data.
3. Extract mesh_extractor.ts Broken Voxel cubes, not Marching Cubes. No interpolation. Root cause of sphere problem.
4. Bake texture_baker.ts Basic Nearest-neighbor. No PBR, no normal maps. Functional but crude.
4b. Animate animation_extractor.ts Basic Center-of-mass trajectory only. Single bone. Limited but correct.

The pipeline's stages 1-2 (compilation and simulation) produce genuinely rich data. The field output from a QCD simulation on a 4D lattice, for example, contains real gauge field configurations. But stages 3-4 collapse this richness into blocky voxels with flat colors. Replacing the mesh extractor with true Marching Cubes (with edge interpolation and normal estimation) would immediately unlock visual diversity from the existing physics.

12. Critical Gaps & Risks

P0 — The Sphere Problem (Asset Pipeline)

The mesh extractor does not implement actual Marching Cubes. It generates identical-looking cubic voxel shells regardless of the input field topology. This is the single biggest blocker to demonstrating the platform's value — users see the same shape for every domain. Fix: Implement true Marching Cubes with edge interpolation and smooth normal estimation.

P0 — Dual Backend Unification

Two backends that each have capabilities the other lacks. The frontend only talks to the TypeScript backend, which means the 26 Python engines, full GSPL parser, gene system, and composition engine are all unreachable from the UI. Fix: Either port the Python modules to TypeScript, or set up the Python backend as a service called by the TS server.

P0 — No Authentication

The API is completely public. Anyone can create, mutate, delete, and sign seeds. No users, no sessions, no permissions. Fix: Implement JWT auth with user accounts.

P1 — Determinism Violation

The spec mandates xoshiro256** RNG for reproducibility, but the TS backend uses Math.random() in mutation fallbacks. This breaks the core promise: same seed → bit-identical artifact. Fix: Implement seeded xoshiro256** in TypeScript and use it everywhere.

P1 — External AI Dependency

Prompt-to-seed generation, intelligent mutation, and embedding all depend on Gemini API calls. If the API is down, rate-limited, or deprecated, core user-facing features degrade to hash-based fallbacks. Fix: Build the native GSPL agent (Llama 3 + RAG) as specified in the vision.

P1 — Frontend Bug

CompositionPanel.jsx references useEffect without importing it. The composition graph won't render on mount. Fix: Add useEffect to the React import.

P2 — No Deployment Infrastructure

No Docker, no CI/CD, no health checks, no structured logging, no monitoring. The README shows manual npm run dev only. Fix: Docker Compose (app + db + redis), GitHub Actions pipeline.

13. Path Forward

The platform's core innovation is genuine and deep. The QFT solvers are real computational physics. The gene system is mathematically complete. The GSPL specification is publication-ready. The 26 engines all contain real logic. The sovereignty layer uses production-grade cryptography. This is not vaporware — it is a functional prototype with a single critical bottleneck (the asset pipeline) and an architectural split (dual backends) that prevents the full vision from being realized in the UI.

Immediate Impact (Week 1-2)

Fix the Sphere Problem

Replace the voxel cube extractor with true Marching Cubes. This single change would make every domain produce visually distinct geometry from the QFT fields. Add smooth normals, basic PBR material properties, and edge interpolation. The physics is already generating rich field data — the visualization just needs to faithfully represent it.

Foundation (Week 2-4)

Unify the Backends

Port the Python GSPL parser, gene system, and composition engine to TypeScript (or expose the Python backend as a microservice). This unlocks all 26 domain engines, the full type system, and cross-domain functor composition in the live UI. Implement the seeded xoshiro256** RNG to restore determinism.

Production (Week 4-8)

Auth + Infrastructure + Native Agent

Add JWT authentication, Docker Compose deployment, CI/CD pipeline, and begin work on the native GSPL agent to eliminate the Gemini dependency. Harden the API with rate limiting, input validation, and structured logging.

Vision (Week 8+)

The Living Creative Operating System

On-chain sovereignty (L2 minting), IPFS/Arweave storage, infinite canvas UI, agent swarms, creative version control (Git for Seeds), WebAssembly engine execution, and the full game generation pipeline. These are achievable extensions of the existing architecture, not rewrites.
